# Meeting Minute App — Architecture & Data Model

> Native Android (Kotlin) first. iOS (Swift) later. No cross-platform.

---

## Stack

| Layer | Tech |
|---|---|
| Mobile (Android) | Kotlin + Jetpack Compose + Room |
| Mobile (iOS) | Swift + GRDB (suggested) |
| Backend | Supabase (Postgres + Auth + Storage) |
| Transcription & Diarization | Assembly AI |
| Summary & Chat | DeepSeek API via Supabase Edge Function |
| Audio Storage | Supabase Storage (S3-backed) |
| Sync | Local-first, last-write-wins, timestamp-based |

---

## Features

- In-app recording + audio file import (future: YouTube video import)
- Offline recording; queue processing when online
- Transcription with speaker diarization
- Summary generation
- Chat scoped to the current meeting only
- Share as Markdown / PDF / Word / Plain text
- Inline bottom-bar audio player with tap-to-seek from transcript timestamps
- Speaker auto-labeling ("Speaker A") with user rename

---

## Auth

- **MVP:** Supabase email auth
- **Later:** Google Sign-In, Apple Sign-In

## Sync Rules (Last Write Wins)

No `sync_status` or `last_sync_at` column. Conflict resolution is a direct comparison of `updated_at` between local and remote.

| Local state | Remote state | Action |
|---|---|---|
| Exists | Missing | Upload local |
| `updated_at` newer | `updated_at` older | Push local |
| `updated_at` older | `updated_at` newer | Pull remote |
| Equal | Equal | Skip |
| `deleted_at` set (newest) | Any | Propagate delete |

**Tie-breaker:** If `updated_at` is exactly equal, local wins (arbitrary but consistent).

**Sync order:**
1. `meetings`
2. `speakers`
3. `transcript_segments`
4. `summaries`
5. `chat_messages`

**Audio file sync logic:**
- `audio_url IS NULL AND local_audio_path IS NOT NULL` → audio file needs upload to Supabase Storage.
- Upload first, then update `audio_url` and `audio_bucket_path`, then sync the row.

**Retry:** Sync worker retries on failure. On next run, the same `updated_at` comparison runs again.

## Supabase Edge Functions

| Function | Input | Output | Side Effects |
|---|---|---|---|
| `transcribe` | `audio_url` | Diarized transcript | Inserts `speakers` + `transcript_segments`; updates `meeting.status` |
| `summarize` | Transcript text | Summary + key points | Inserts `summaries`; updates `meeting.status` |
| `chat` | Transcript text + chat history + new user message | Assistant reply | Inserts assistant `chat_messages` |

API keys (Assembly AI, DeepSeek) live in Edge Function secrets. Never client-side.

## Screen & Navigation

- **Home:** List of meetings
- **Meeting Detail:** Bottom nav with 3 tabs
  - **Transcript:** Scrollable segments with speaker labels, timestamps, tap-to-seek
  - **Summary:** Rendered summary + key points
  - **Chat:** Message list with user input
- **Share Sheet:** User picks format (Markdown / PDF / Word / Plain text)
- **Player:** Bottom bar on Meeting Detail, scrubs when transcript timestamp tapped

## Decisions Log

| Date | Decision | Rationale |
|---|---|---|
| 2026-06-06 | Native Kotlin first, Swift later | No Mac yet; avoid cross-platform lock-in |
| 2026-06-06 | Room (Android), GRDB suggested (iOS) | Room is Android standard; GRDB is closest Swift equivalent |
| 2026-06-06 | No `sync_status` or `last_sync_at` columns | Direct `updated_at` comparison between local and remote is sufficient |
| 2026-06-06 | Soft deletes via `deleted_at` | Safer sync; easy to undo |
| 2026-06-06 | Chat scoped to single meeting only | Simpler context window; no global RAG needed |
| 2026-06-06 | Share formats: Markdown, PDF, Word, Plain text | Let user choose per share |
| 2026-06-06 | Inline bottom-bar player | Fastest UX; no full-screen context switch |
| 2026-06-06 | Auto speaker labels + manual rename | Assembly AI gives "Speaker A"; user maps to real names after |
| 2026-06-06 | `profiles` table in public schema | Mirrors `auth.users` for `display_name`, `avatar_url`; `auth.users` is auto-generated |
